# コードレビュー改善メモ

## 1. コードの重複 [HIGH]

**`fetchMembers` が二重呼び出しされている**

`useChannelMembership.ts` の `handleJoin`/`handleLeave` 内で `fetchMembers()` を呼び、さらに `App.tsx` の `refreshAfterMembership` でも `fetchMembers()` を呼んでいる。参加/退出のたびに同じAPIが2回実行される。

→ どちらか一方に統一する

---

## 2. 型定義の不足 [HIGH]

**Supabaseクエリ結果が `any` 相当で型安全でない**

`useChannels.ts` 等で `c.id as string` のように `as` キャストで型を付けている。Supabaseの Database 型定義が未導入のため、テーブルのカラム名を間違えてもコンパイルエラーにならない。

→ `supabase gen types typescript` で型を生成し `createClient<Database>(...)` を使う

---

## 3. 命名の改善 [MEDIUM]

**`handleEdit`/`handleDelete` が実態と乖離している**

名前は「編集・削除を実行する」ことを示唆するが、実際にはローカルstateのみ操作しておりDBに永続化されない。リロードすると変更が消失する。

→ `updateMessageLocally`/`removeMessageLocally` に改名するか、Supabaseへの永続化を追加する

---

## 4. パフォーマンスの問題 [MEDIUM]

**メッセージ一覧が毎回全件再レンダリングされる**

リアクション1つ追加するだけで `messages` 配列全体が更新され、全メッセージ行が再レンダリングされる。各行にはAvatar、Popover等の複雑な要素がある。

→ 個別メッセージを `React.memo` の `MessageItem` コンポーネントに抽出する

---

## 5. エラーハンドリングの不足 [CRITICAL]

**Supabase環境変数が未設定でも起動時に検知できない**

`supabase.ts` で `as string` キャストしているため、`.env` がなくても `undefined` のまま動いてしまい、API呼び出し時に不明瞭なエラーになる。

→ 起動時に存在チェックして、未設定なら即座にエラーを投げる

---

## 修正履歴

【修正1】修正前: `useChannelMembership.ts`の`handleJoin`/`handleLeave`内と`App.tsx`の`refreshAfterMembership`内で`fetchMembers()`が二重に呼ばれていた / 修正後: フック側の`fetchMembers()`呼び出しを削除し、`App.tsx`の`refreshAfterMembership`に一本化した / 理由: 参加/退出のたびに同じAPIが2回実行される無駄を解消し、データ取得の流れを1箇所に集約して保守性を向上させるため

【修正2】修正前: `handleEdit`/`handleDelete`という命名がDB永続化を示唆していたが、実際はローカルstate操作のみだった / 修正後: `updateMessageLocally`/`removeMessageLocally`にリネームし、props名も`onEditLocal`/`onDeleteLocal`に統一した / 理由: 関数名と実際の動作の乖離をなくし、リロードで消える操作であることを名前から明確に伝えるため

【修正3】修正前: `supabase.ts`で環境変数を`as string`でキャストしており、未設定でも`undefined`のまま動いて不明瞭なランタイムエラーになっていた / 修正後: `as string`を削除し、起動時に存在チェックを追加して未設定なら即座に明確なエラーメッセージを投げるようにした / 理由: `.env`の設定漏れを起動直後に検知できるようにし、デバッグ時間を短縮するため
